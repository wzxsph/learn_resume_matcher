# 06 - 前端架构：Resume Matcher 的 Next.js 16 + React 19 实现

> 本篇聚焦于 `apps/frontend` 目录,系统化梳理其技术栈选型、目录结构、路由组织、配置文件、状态管理与样式系统,帮助读者理解整个前端工程的骨架。

## 一、整体技术栈

Resume Matcher 前端是单仓多包 (monorepo) 中独立运行的 Next.js 应用,版本与工具链如下:

| 维度 | 选型 | 版本 | 关键作用 |
|------|------|------|----------|
| 框架 | Next.js | 16.2.6 | App Router + Turbopack |
| 视图层 | React | 19.2.4 | Server Components、use Hooks |
| 语言 | TypeScript | 5 (strict) | 类型安全 |
| 样式 | Tailwind CSS | v4 | CSS-first 配置,无 `tailwind.config.js` |
| 富文本 | Tiptap | 3.20 | 简历编辑 |
| 拖拽 | @dnd-kit | 6/10/3 | 章节排序 |
| 测试 | Vitest + Testing Library | 4.0 | jsdom 环境 |
| 图标 | lucide-react | 0.575 | 通过 deep path 直接 import 避免 barrel |

`apps/frontend/package.json:13-29` 列出完整依赖。`apps/frontend/package.json:5-12` 定义脚本:

```json
"dev": "next dev --turbopack",
"build": "next build",
"lint": "eslint .",

"test": "vitest run"
```

注意 `dev` 使用 Turbopack 启动,`build` 仍然使用 `next build`(默认 Webpack),二者都启用 App Router。

## 三、路由组织

### 3.1 路由组 `(default)`

`app/(default)/` 是 Next.js 16 App Router 的路由组 (括号包裹的目录不会出现在 URL 中)。`apps/frontend/app/(default)/layout.tsx:1-19` 将所有受 Provider 包裹的页面收纳在同一棵布局树下:

```tsx
<StatusCacheProvider>
  <LanguageProvider>
    <ResumePreviewProvider>
      <LocalizedErrorBoundary>
        <main className="min-h-screen flex flex-col">{children}</main>
      </LocalizedErrorBoundary>
    </ResumePreviewProvider>
  </LanguageProvider>
</StatusCacheProvider>
```

`StatusCacheProvider` 负责缓存 `/status`(`apps/frontend/lib/context/status-cache.tsx:39-216`),`LanguageProvider` 维护 UI/内容语言(`apps/frontend/lib/context/language-context.tsx:26-113`),`ResumePreviewProvider` 与 `LocalizedErrorBoundary` 分别来自 `components/common/`。`apps/frontend/app/layout.tsx:5-15` 在根布局加载 Geist + Space Grotesk 字体并设置 `<html lang="en-US">`,是整棵布局树的最外层。

### 3.2 页面清单

| 路由 | 文件 | 渲染方式 | 用途 |
|------|------|----------|------|
| `/` | `app/(default)/page.tsx:3-5` | Server | 落地页,直接渲染 `<Hero/>` |
| `/dashboard` | `app/(default)/dashboard/page.tsx:33-541` | Client | 主面板,resume 列表、上传、删除、重试、状态卡 |
| `/builder` | `app/(default)/builder/page.tsx:3-5` | Server 包装 | 委托给 `components/builder/resume-builder.tsx` |
| `/tailor` | `app/(default)/tailor/page.tsx:24-504` | Client | 粘贴 JD → 触发 improve preview → 弹 diff modal |
| `/settings` | `app/(default)/settings/page.tsx:104-1350` | Client | LLM 配置、特性开关、提示词、Danger Zone |
| `/resumes/[id]` | `app/(default)/resumes/[id]/page.tsx` | Client | 单个简历查看、下载、改名、AI 增强 |

### 3.3 动态路由 `[id]`

`/resumes/[id]` 与 `/print/*` 都使用 Next.js 16 的 `Promise<{...}>` 形态参数。`apps/frontend/app/print/resumes/[id]/page.tsx:17-38` 给出了示例:

```ts
type PageProps = {
  params: Promise<{ id: string }>;
  searchParams?: Promise<{ template?: string; pageSize?: string; ... }>;
};
```

Next.js 15+ 改为异步 `params`/`searchParams`,所有页面必须用 `await` 解析。

### 3.4 服务端 `print/*` 子树

`apps/frontend/CLAUDE.md:26` 明确指出:打印页是**有意保留**的 Server Component。它们没有 Provider 包裹,通过 `API_BASE` + `lib/i18n/server.ts:translate` 直接在后端渲染并交给 headless Chromium 转 PDF。这种"分叉的 App Router 子树"是 Next.js 16 一个非常实用的模式。

## 四、关键配置文件解析

### 4.1 `next.config.ts`

`apps/frontend/next.config.ts:1-45` 的核心要点:

1. **环境变量决定后端地址**:`BACKEND_ORIGIN` 默认为 `http://127.0.0.1:8000`(`next.config.ts:3`)。
2. **standalone 输出**:`output: 'standalone'`(`next.config.ts:6`)启用独立部署模式,适合 Docker。
3. **超时与 tree-shaking**:`experimental.proxyTimeout: 240_000`(`next.config.ts:8`)与 `optimizePackageImports`(`next.config.ts:9-19`)联合把 lucide/tiptap/dnd-kit 拆解为按需 import,缩短冷启动 200~800ms。
4. **rewrites 代理**:`next.config.ts:21-42` 将 `/api/*`、`/docs`、`/redoc`、`/openapi.json` 全部代理到 FastAPI 后端。注释 `next.config.ts:22-23` 警告:"filesystem 路由优先",禁止创建 `app/api/*` 路由,否则会遮蔽代理。

### 4.2 `tsconfig.json`

`apps/frontend/tsconfig.json:1-33` 启用 `strict: true`、`moduleResolution: "bundler"`、`jsx: "react-jsx"`、路径别名 `@/*`。`include` 字段(`tsconfig.json:25-31`)涵盖 `.next/dev/types/**/*.ts`,这意味着 Next.js 16 的 dev 类型(基于 `typedRoutes`)会自动被纳入编译。

### 4.3 Tailwind v4 与 PostCSS

Tailwind v4 取消了 `tailwind.config.js`,改为 CSS-first 配置。`apps/frontend/postcss.config.mjs:1-7` 仅注册 `@tailwindcss/postcss` 插件:

```js
export default { plugins: { '@tailwindcss/postcss': {} } };
```

真正的 token 来自 `app/(default)/css/globals.css` 的 `@theme inline {}` 块(由 `app/layout.tsx:3` import)。`apps/frontend/CLAUDE.md:117-129` 列出 Swiss Design token:

| Token | 值 | Tailwind 类 |
|-------|----|------------|
| Canvas | `#F0F0E8` | `bg-background` `bg-canvas` |
| Ink | `#000000` | `text-ink` `text-ink-soft` |
| Hyper Blue | `#1D4ED8` | `text-primary` `bg-primary` |
| Signal Green | `#15803D` | `text-success` |
| Alert Orange | `#F97316` | `text-warning` |
| Alert Red | `#DC2626` | `text-destructive` |

规范要求 `rounded-none` 全局生效、1px 黑边、硬阴影 `shadow-sw-xs..xl`(实心墨色无模糊)。`apps/frontend/CLAUDE.md:128` 强调"硬阴影是 Swiss 设计的特征"。

### 4.4 ESLint 与 Prettier

`apps/frontend/eslint.config.mjs:1-35` 合并了 `eslint-config-next/core-web-vitals`、`/typescript`、`eslint-config-prettier`,并在 `rules`(`eslint.config.mjs:25-29`)中关闭 `react-hooks/set-state-in-effect` —— 注释说明现有代码依赖 effect 中 setState 同步 props 与 DOM 测量。Prettier 通过 ESLint 的 `prettier/prettier: 'error'` 强制。

### 4.5 Vitest

`apps/frontend/vitest.config.ts:1-21` 配置 jsdom 环境、`@vitejs/plugin-react`、`./vitest.setup.ts`,并复用 `@/*` 别名。`vitest.setup.ts` 注入 `@testing-library/jest-dom`。

## 五、状态管理方案

项目没有引入 Redux/Zustand/Jotai,只使用 React Context + 自定义 Hooks,具体分层如下:

| 层 | 来源 | 关注点 |
|----|------|--------|
| 全局只读缓存 | `StatusCacheProvider`(`lib/context/status-cache.tsx:39-216`) | 30 min 自动刷新 `/status`、乐观计数器 `incrementResumes` 等 |
| 双向同步 | `LanguageProvider`(`lib/context/language-context.tsx:26-113`) | UI 语言走 localStorage,内容语言走 backend |
| 预览数据 | `ResumePreviewProvider`(`components/common/resume_previewer_context.tsx`) | `/tailor` 写入、`/builder` 读取 |
| 流程状态 | 自定义 Hook(`hooks/use-enrichment-wizard.ts`、`hooks/use-regenerate-wizard.ts`) | useReducer 或 useState 局部管理 |
| 本地存储 | `localStorage`(`master_resume_id`、`resume_builder_draft` 等) | 客户端持久化 |

`apps/frontend/CLAUDE.md:77-79` 写明:"StatusCache is React Context, not a fetch lib",这是项目有意选择 Context 的原因 —— Context 是稳定订阅通道,而 React Query/SWR 等数据层库会引入额外的失效/重试策略,与 30min 缓存粒度不匹配。

## 六、样式与设计系统

`apps/frontend/CLAUDE.md:112-130` 强制要求所有 UI 走 Swiss International Style:

- **字体**:标题用 `font-serif`,正文用 `font-sans` (Geist),元数据用 `font-mono` (Space Grotesk)。`app/layout.tsx:5-15` 通过 `next/font/google` 注入 CSS 变量。
- **形状**:`rounded-none` 全局生效,无圆角。
- **边框与阴影**:1px 黑色实边 + 硬阴影(`shadow-sw-*`),无模糊、无渐变。
- **图标**:lucide-react,深路径 import(如 `lucide-react/dist/esm/icons/loader-2`),避免 barrel re-export 拖慢打包。`next.config.ts:9-19` 同步配置 `optimizePackageImports` 作为兜底 tree-shake。
- **打印**:`components/resume/styles/*.module.css` 自带 `_tokens.css`、`swiss-single`、`swiss-two-column`、`modern` 等 CSS Module,专供 print 路由使用。

## 九、关键设计取舍

1. **Turbopack dev + Webpack build**:Turbo 提升 HMR 速度,但 `next build` 仍由 Webpack 执行,这意味着线上打包路径与开发路径略有不同,遇到 issue 时要分头排查。
2. **Client 优先、Print 例外**:`(default)/layout.tsx` 用 Provider 树包装,而 `print/*` 全部保持 Server 渲染,便于 headless Chromium 准确抓取。这是一种"局部 Server、局部 Client"的分叉。
3. **i18n 必须结构对齐**:`lib/i18n/messages.ts:11-17` 把所有 locale 强制 cast 为 `Messages = typeof en`,任何对 `en.json` 的增删都必须镜像到其他 4 个文件,否则 `npm run build` 失败。
4. **路径别名 + 单一 client.ts**:`lib/api/client.ts:43-74` 的 `apiFetch` 是 240s 超时 + 友好错误的统一封装,所有页面与 Hook 都通过它访问后端,避免在组件里直接 `fetch('/api/...')`。
5. **本地存储是 Source of Truth 的一部分**:`master_resume_id`、`resume_builder_draft` 等 localStorage 字段(`apps/frontend/CLAUDE.md:97-102`)是约定俗成的"客户端主键",删除/重置时需主动清理(见 `app/(default)/settings/page.tsx:536-541` 的 `handleResetDatabase`)。
