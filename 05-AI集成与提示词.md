# Resume Matcher 的 AI 集成与提示词工程实战

> 本教程面向希望把 LLM 接入真实生产系统的 Python 后端开发者。我们以 Resume Matcher（一个面向求职场景的 AI 改写简历服务）为蓝本，从 LiteLLM 封装、提示词模板设计、端到端工作流、容错策略、新增 provider 的步骤等角度逐项拆解。所有结论都基于 `apps/backend/app/` 下的实际源码，并标注 `文件:行号` 以便复核。

---

## 1. 项目中的 LLM 集成位置一览

Resume Matcher 并不把 LLM 调用散落在路由层，而是统一收敛到三层：

| 层 | 文件 | 职责 |
| --- | --- | --- |
| 传输/容错层 | `apps/backend/app/llm.py` | 适配 LiteLLM，处理多 provider、重试、超时、JSON 抽取 |
| 提示词层 | `apps/backend/app/prompts/__init__.py`、`templates.py`、`enrichment.py`、`refinement.py` | 全部提示词模板 + 占位符校验 |
| 业务编排层 | `apps/backend/app/services/` | `parser.py`（解析）、`improver.py`（diff 改写）、`refiner.py`（多轮打磨）、`cover_letter.py`（求职信/外联/标题） |

路由层（`apps/backend/app/routers/`）只负责：接收请求 → 调服务 → 返回 Pydantic 模型。这种分层让 LLM 相关能力（切换 provider、调整 prompt、加重试）可以在不动路由的情况下集中迭代。

## 2. LiteLLM 封装分析（`apps/backend/app/llm.py`）

### 2.1 支持的 provider

`llm.py:302-311` 通过 `_PROVIDER_KEY_MAP` 显式列举了 8 个 provider：

| provider 字符串 | LiteLLM 实际路由 | 典型模型 |
| --- | --- | --- |
| `openai` | 不加前缀 | gpt-5、gpt-4o |
| `openai_compatible` | `openai/` 前缀 | llama.cpp / vLLM / LM Studio |
| `anthropic` | `anthropic/` 前缀 | Claude Haiku 4.5、Claude Opus 4.x |
| `gemini` | `gemini/` 前缀 | Gemini 3 Flash |
| `openrouter` | `openrouter/` 前缀 | 任何 OpenRouter 列表里的模型 |
| `deepseek` | `deepseek/` 前缀 | DeepSeek Chat、DeepSeek R1 |
| `groq` | `groq/` 前缀 | Llama 3 在 Groq 上的部署 |
| `ollama` | `ollama_chat/` 前缀 | 本地 ollama 服务 |

注意 `get_model_name` (`llm.py:403-448`) 做了两件事：先按 `provider` 表加前缀，再识别 `openrouter/anthropic/...` 之类的"已经带前缀"的形式避免重复拼接。`ollama` 特殊用 `ollama_chat/` 而不是 `ollama/`，因为 chat 端点支持多轮消息数组（`llm.py:421`）。

### 2.2 `api_base` 的归一化

本地自托管（llama.cpp、vLLM、Ollama）和 OpenAI 兼容代理经常出现 `http://host:8080/v1` 这种 URL。`llm.py:64-114` 的 `_normalize_api_base` 做了三件事：

1. `strip + rstrip('/')`；
2. 对 Anthropic / Gemini / OpenRouter / Ollama，把末尾的 `/v1` 去掉（LiteLLM 内部还会拼一次，否则会出现 `/v1/v1/...` 的 404，参见 `llm.py:91-112` 注释）；
3. 对 OpenAI / OpenAI 兼容端点，原样保留 URL（包括 `/v1`），因为 OpenAI 客户端自己会正确解析。

`llm.py:617-620` 还用 `_scrub_secrets` 在向上游报告错误时遮蔽 `sk-...`、`AIza...`、`Bearer ...` 等敏感串，避免 Settings UI 间接读回 API key。

### 2.3 API key 的解析与防泄漏

`llm.py:323-349` 的 `resolve_api_key` 实现了"单一真实来源"，所有需要 key 的代码路径都必须走它。优先级是：

1. `stored["api_key"]`（顶层字段，UI 写）；
2. `stored["api_keys"][provider]`（按 provider 分桶）；
3. `settings.llm_api_key`（环境变量）。

但对 `openai_compatible` 与 `ollama` 这两个本地 provider，`llm.py:318-320` 的 `_PROVIDERS_WITHOUT_ENV_KEY_FALLBACK` 集合强制跳过环境变量——因为 `LLM_API_KEY` 通常是付费 OpenAI 密钥，绝不能误传到本地明文服务。`openai_compatible` 用户留空时还会注入哨兵值 `sk-no-key` 满足 OpenAI 客户端的"非空字符串"校验（`llm.py:120-131`）。

### 2.4 Router、RetryPolicy 与 cooldown

`llm.py:472-504` 用 `litellm.Router` 包装一个单 deployment 的模型，配置如下：

- `num_retries=3`：LiteLLM 在 transport 层最多自动重试 3 次；
- `RetryPolicy` 按错误类型差异化：401/400/内容策略违规 → 0 次（重试无意义），超时/500 → 2 次，429 → 3 次；
- `disable_cooldowns=True`：因为只有一个 deployment，cooldown 会让后端在瞬时故障后整体黑屏。

`get_router` (`llm.py:507-526`) 用 `provider|model|hash(api_key)|api_base` 指纹缓存 Router，仅在配置变化时重建，配合 `threading.Lock` 处理并发首请求。

### 2.5 能力探测：能力来自 Registry 而不是硬编码

避免在代码里写 `if provider == "openai"` 这类硬编码——`llm.py` 通过 `litellm.get_model_info` 查询注册表：

- `_supports_json_mode` (`llm.py:702-731`)：判断模型是否在 `supported_openai_params` 里含 `response_format`。Ollama 走 `format="json"`，所以特判返回 True。
- `get_safe_max_tokens` (`llm.py:736-778`)：把调用方请求的 token 数 clamp 到 `max_output_tokens`，未知模型 fallback 4096。
- `_supports_temperature` (`llm.py:827-876`)：返回 False 时调用方就不会传 `temperature`，避免 BadRequestError。同时为 Anthropic Claude Opus 4.x（已弃用 temperature）、Moonshot Kimi K2.6（只允许 1.0）做硬编码特判。
- 顶层 `litellm.drop_params = True` (`llm.py:31`)：让 LiteLLM 主动丢弃目标 provider 不识别的参数（如 `reasoning_effort`、非默认 `temperature`），不再抛 `UnsupportedParamsError`。

### 2.6 两条主调用 API

`complete` (`llm.py:651-699`) 与 `complete_json` (`llm.py:1026-1156`) 是服务的唯二对外函数。区别：

| 维度 | `complete` | `complete_json` |
| --- | --- | --- |
| 输出 | 纯文本 | 解析后的 `dict` |
| JSON 模式 | 否 | 自动启用 `response_format={"type":"json_object"}` |
| 应用层重试 | 0 | 2 次，针对 malformed JSON / truncation |
| 重试策略 | 全部由 Router 处理 | 解析失败时改温度（0.1→0.3→0.5→0.7，参见 `_get_retry_temperature` `llm.py:879-894`） |
| 失败兜底 | 抛 `ValueError` | 失败时关闭 JSON 模式，改用 prompt 强制（`json_mode_failed` 标志，`llm.py:1126-1133`） |

`complete_json` 还有 schema 感知的截断检测 `_appears_truncated` (`llm.py:781-824`)：当 schema 是 `resume` 且 `workExperience/education/skills` 出现空数组、`enrichment` 缺 `items_to_enrich/questions` 时判为截断并重试。

### 2.7 错误处理与安全

`llm.py:693-699` 是 LLM 调用的"安全降级"模板：

```python
except Exception as e:
    logging.error(f"LLM completion failed: {e}", extra={"model": model_name})
    raise ValueError(
        "LLM completion failed. Please check your API configuration and try again."
    ) from e
```

详情在服务端日志，客户端只拿到一段通用提示。`_scrub_secrets` (`llm.py:286-299`) 在 `error_detail` 露出前再次擦除 key。

`check_llm_health` (`llm.py:529-648`) 把 `404 /v1/v1/`、`404` 通用、HTML 响应等常见错误映射到独立 `error_code`，前端可以直接用来定位"用户填的 base URL 多带了一个 `/v1`"。

## 3. 提示词模板设计

### 3.1 文件分布

| 文件 | 主要常量 | 备注 |
| --- | --- | --- |
| `apps/backend/app/prompts/__init__.py:1-57` | 集中 re-export 与 `validate_prompt_placeholders` | 业务代码只 `from app.prompts import ...` |
| `apps/backend/app/prompts/templates.py` | 解析、关键词、3 个改写、diff、skill plan、求职信、外联、标题 | 大本营 |
| `apps/backend/app/prompts/enrichment.py` | `ANALYZE_RESUME_PROMPT`、`ENHANCE_DESCRIPTION_PROMPT`、`REGENERATE_ITEM_PROMPT`、`REGENERATE_SKILLS_PROMPT` | 与弱描述改写相关 |
| `apps/backend/app/prompts/refinement.py` | `KEYWORD_INJECTION_PROMPT`、`VALIDATION_POLISH_PROMPT`、AI 短语黑/白名单 | 多轮打磨 |

### 3.2 模板编写约束

`apps/backend/app/prompts/templates.py` 几乎所有 prompt 都遵守同一规则——**JSON 输出模板中的 `{}` 必须写成 `{{}}`，否则 `str.format()` 会抛 `KeyError`**。`EXTRACT_KEYWORDS_PROMPT` (`templates.py:179-196`)、`DIFF_IMPROVE_PROMPT` (`templates.py:457-527`)、所有 `enrichment.py` 里的 prompt 都是如此；`PARSE_RESUME_PROMPT` (`templates.py:154-177`) 是例外，它把 JSON 模板通过 `{schema}` 整体注入，因此不需要转义。

`templates.py:4-15` 维护 `LANGUAGE_NAMES` 字典，把 `en / es / zh / ja / pt` 映射到完整语言名，并提供 `get_language_name(code)` (`templates.py:13-15`)。`{output_language}` 占位符在所有 prompt 中都接收完整语言名（如 "Brazilian Portuguese"），而不是语言码。

### 3.3 三种改写策略

`templates.py:320-342` 定义 `IMPROVE_RESUME_PROMPTS = {nudge, keywords, full}`，每种对应不同 `CRITICAL_TRUTHFULNESS_RULES`（`templates.py:198-227`）：

| prompt_id | 风格 | truthfulness 第 7 条 |
| --- | --- | --- |
| `nudge` | 最小改动，只重述现有内容 | "DO NOT add new bullet points or content" |
| `keywords` | 在重述里织入关键词 | "You may rephrase existing bullet points to include keywords" |
| `full` | 可扩展/补充 | "You may expand existing bullet points or add new ones that elaborate" |

`DEFAULT_IMPROVE_PROMPT_ID = "keywords"` (`templates.py:344`)。`apps/backend/app/routers/resumes.py:67-71` 的 `_get_default_prompt_id` 从 `config.json` 读用户偏好，未设置时回落到 `keywords`。

### 3.4 diff 改写：让 LLM 返回"动作"而非全文

`DIFF_IMPROVE_PROMPT` (`templates.py:457-527`) 与三选一全文改写相比，关键差异是它只允许 LLM 输出 `changes` 数组，每条 change 都带 `path / action / original / value / reason`。`path` 仅限 4 类白名单（`apps/backend/app/services/improver.py:80-85`）：

```python
_ALLOWED_PATH_PATHS = [
    re.compile(r"^summary$"),
    re.compile(r"^workExperience\[\d+\]\.description(\[\d+\])?$"),
    re.compile(r"^personalProjects\[\d+\]\.description(\[\d+\])?$"),
    re.compile(r"^additional\.technicalSkills$"),
]
```

而 `personalInfo / customSections / years / company / institution` 等敏感字段被列入黑名单（`improver.py:88-135`）。`apply_diffs` 在落库前要过 4 道门：

1. path 在白名单；
2. path 不在黑名单；
3. path 实际能 resolve 到原值；
4. `replace` 类动作必须 `original` 文本与实际内容一致（大小写不敏感比较，见 `_verify_original_matches`，`improver.py:205-211`）。

这就把 LLM "幻觉式重写"风险压到最低——就算模型给出"把 Apple 改成 Google"这种伪造信息，也会被原始匹配失败这一关挡掉。

### 3.5 富集（enrichment）四件套

`apps/backend/app/prompts/enrichment.py` 维护 4 个 prompt，结构非常工整：

- `ANALYZE_RESUME_PROMPT`（`enrichment.py:3-82`）：让模型审视简历，输出"待补强条目 + 6 个以内的引导问题"，每题附 `placeholder` 提示。
- `ENHANCE_DESCRIPTION_PROMPT`（`enrichment.py:84-125`）：基于用户回答，**追加**新 bullet，而不是替换；prompt 显式写 "ADD new bullet points … DO NOT rewrite or replace existing bullets"。
- `REGENERATE_ITEM_PROMPT`（`enrichment.py:133-172`）：整段重写，约束"绝不引入新事实，没有数字就强调定性影响"。
- `REGENERATE_SKILLS_PROMPT`（`enrichment.py:175-195`）：针对技能列表的重写。

四份 prompt 都强制"输出 JSON only"，并通过 `system_prompt` 二次强调。`enrichment.py:5`、`86`、`135`、`177` 都用 `{output_language}` 引导所有生成文本走多语言。

### 3.6 自定义 prompt：占位符必须存在

`apps/backend/app/prompts/__init__.py:22-26` 定义了求职信/外联类 prompt 的强制占位符：

```python
REQUIRED_FEATURE_PROMPT_PLACEHOLDERS = (
    "{job_description}", "{resume_data}", "{output_language}",
)
```

`validate_prompt_placeholders` (`__init__.py:29-39`) 在 `PUT /config/feature-prompts` 写入前校验，缺失就 422。`apps/backend/app/services/cover_letter.py:17-33` 的 `_resolve_feature_prompt` 在运行时再兜一次：自定义 prompt `.format()` 失败就回落默认模板并 `logging.warning`，避免磁盘被手工改坏导致整个生成崩溃。

## 4. 端到端工作流

下面把 `POST /api/v1/resumes/improve/preview` 走通，看 LLM 在每个环节如何被调用。

### 4.1 上传与解析

`apps/backend/app/routers/resumes.py:511-594` 的 `upload_resume`：

1. 校验类型（PDF/DOC/DOCX，<=4 MB，`resumes.py:503-509`）；
2. `parse_document` (`apps/backend/app/services/parser.py:119-141`) 用 `markitdown` 把文件转 Markdown；
3. `parse_resume_to_json` (`parser.py:144-176`) 用 `complete_json` + `PARSE_RESUME_PROMPT` 让 LLM 抽结构化 JSON；
4. `restore_dates_from_markdown` (`parser.py:40-116`) 把 markdown 里出现的 `Jan 2020 - Dec 2023` 反向补回被 LLM 误删月份的 year-only 字段；
5. `ResumeData.model_validate` 做 Pydantic 二次校验。

### 4.2 关键词抽取

`apps/backend/app/services/improver.py:555-572` 的 `extract_job_keywords` 用 `EXTRACT_KEYWORDS_PROMPT`，产出 `required_skills / preferred_skills / experience_requirements / education_requirements / key_responsibilities / keywords / experience_years / seniority_level` 字典。`routers/resumes.py:728-749` 用 `job_id` + `job_keywords_hash` 缓存，重复 tailor 不必二次 LLM 调用。

### 4.3 改写主流程（diff 模式）

`routers/resumes.py:719-960` 的 `_improve_preview_flow` 是核心：

1. **Skill 计划** (`improver.py:785-835`)：先 `generate_skill_target_plan` 问 LLM "这份简历要强调哪些技能"，再用 `verify_skill_target_plan` (`improver.py:705-782`) 把建议归为 accepted/rejected；只有 `existing` / `jd_added` / `supported_by_resume` 三个 source 才允许注入。
2. **生成 diff** (`improver.py:457-552`)：用 `DIFF_IMPROVE_PROMPT` 让 LLM 只输出 `changes[]`。
3. **应用 diff** (`improver.py:214-361`)：四道门 + `apply_diffs`；不合规的 change 全部进 `rejected_changes` 列表返回给前端。
4. **质量校验** (`improver.py:381-454`)：`verify_diff_result` 检查 section 数量、身份字段、字数膨胀率、新增 metrics 是否在原文中存在。
5. **本地安全网** (`routers/resumes.py:832-844`)：`_preserve_personal_info` / `_restore_original_dates` / `restore_dates_from_markdown` / `_preserve_original_skills` / `_protect_custom_sections`——即便 diff 通过，也强制把"个人姓名 / 关键日期 / 原始技能"按原值回填。

### 4.4 多轮打磨

`refine_resume` (`apps/backend/app/services/refiner.py:76-178`) 串联三步：

1. **关键词注入** (`refiner.py:452-512`)：把 `injectable_keywords` 通过 `KEYWORD_INJECTION_PROMPT` (`apps/backend/app/prompts/refinement.py:138-159`) 织入现有 bullet。`_validate_resume_structure` (`refiner.py:429-449`) 在落库前再做一次结构校验。
2. **AI 短语清除** (`refiner.py:233-287`)：本地 `re.sub` 把 `AI_PHRASE_BLACKLIST`（如 "spearheaded"、"synergy"、"utilized"）替换为 `AI_PHRASE_REPLACEMENTS`（如 "led"、"collaboration"、"used"），但跳过在 JD 文本中出现的短语以免误伤行业术语。
3. **Master 对齐校验** (`refiner.py:290-408`)：检查 `technicalSkills / certificationsTraining / company` 三个字段在 master 中存在，否则按 `critical` violation 处理；`fix_alignment_violations` (`refiner.py:515-562`) 删除虚构内容。

### 4.5 求职信/外联/标题

`apps/backend/app/services/cover_letter.py`：

- `generate_cover_letter` (`cover_letter.py:36-87`)：用 `COVER_LETTER_PROMPT`（`templates.py:349-371`），要求 100-150 词、3-4 段、自定义 prompt 可覆盖。
- `generate_outreach_message` (`cover_letter.py:90-136`)：70-100 词，更口语化。
- `generate_resume_title` (`cover_letter.py:139-168`)：用 `temperature=0.3, max_tokens=60` 让模型严格按 "Role @ Company" 格式输出。

`routers/resumes.py:446-498` 的 `_generate_auxiliary_messages` 用 `asyncio.gather` 并行触发，单点失败不影响其它（异常被收集进 `warnings`）。

### 4.6 富集（enrichment）支线

`apps/backend/app/routers/enrichment.py:87-169` 的 `analyze_resume` 是入口：

- 拉简历 → `ANALYZE_RESUME_PROMPT` 拿 `items_to_enrich / questions / analysis_summary`；
- 单题 180 秒超时（`enrichment.py:118-121`），失败分别映射 504 / 422 / 500。
- 用户回答后 `/enhance` (`enrichment.py:172-325`) 按 `item_id` 聚合答案，**fast path** 直接从 `processed_data` 取条目信息（`enrichment.py:43-84`），避免重新跑 LLM 分析。
- `/regenerate` (`enrichment.py:487-544`) 用 `asyncio.gather` 并行处理多个条目，单条失败只记入 `errors` 列表，不阻塞其它。

### 4.7 时间线总结

| 阶段 | 入口 | LLM 角色 | 输出校验 |
| --- | --- | --- | --- |
| 上传 | `POST /resumes/upload` | `parse_resume_to_json` | `ResumeData` + markdown 日期回填 |
| 富集 | `POST /enrichment/analyze/{id}` | `ANALYZE_RESUME_PROMPT` | Pydantic |
| 改写 | `POST /resumes/improve/preview` | `EXTRACT_KEYWORDS_PROMPT` → `SKILL_TARGET_PLAN_PROMPT` → `DIFF_IMPROVE_PROMPT` | diff 4 道门 + 安全网 + `refine_resume` |
| 确认 | `POST /resumes/improve/confirm` | 不直接调 LLM，但调用 `_generate_auxiliary_messages` | hash 一致性 + `personalInfo` 不变 |
| 收尾 | `POST /resumes/{id}/generate-cover-letter` 等 | `COVER_LETTER_PROMPT` / `OUTREACH_MESSAGE_PROMPT` / `GENERATE_TITLE_PROMPT` | prompt 校验 + 异常降级 |

## 5. 错误处理与重试机制

| 层 | 机制 | 触发条件 | 行为 |
| --- | --- | --- | --- |
| 传输层 | `Router.num_retries=3` + `RetryPolicy` | 429 / 500 / Timeout | LiteLLM 内部指数退避 |
| 应用层 | `complete_json(retries=2)` | JSON 解析失败 / 截断 | 改温度重试 + 关闭 JSON mode 兜底 |
| 业务层 | `asyncio.wait_for(timeout=...)` | LLM 拖死 | 路由层返回 504 |
| 业务层 | 异常日志 + 通用 500 | 未知 | `logger.error(...)` + `raise HTTPException(500, "… Please try again.")` |
| 安全层 | `_sanitize_user_input` | JD 文本含 `ignore previous instructions` 等 | 替换为 `[REDACTED]`（`apps/backend/app/services/improver.py:48-56`） |
| 提示层 | `{critical_truthfulness_rules}` | 全文改写 prompt | 强制 "DO NOT add new skill ..." |
| 数据层 | `_validate_resume_structure` / `_preserve_personal_info` / `_restore_original_dates` | LLM 返回值结构错误 | 局部回退到原值 |

容错策略的核心思路是：**把"内容质量"问题留在应用层，把"传输"问题交给 Router**。`llm.py:1150-1154` 的注释明确写 "Transport errors — Router already retried with backoff. Cooldowns are disabled ... no additional retry is attempted here"。

`apps/backend/app/services/refiner.py:411-426` 的 `_prepare_job_description` 把超过 2000 字符的 JD 截断并打 warning，防止 prompt 过长被 provider 直接拒收。`MAX_JSON_CONTENT_SIZE = 1 MB` (`llm.py:45`) 防止 JSON 抽取时无限递归或被巨大响应拖死。

## 6. 如何添加新的 LLM provider

假设要加一个 `mistral` provider，步骤如下：

1. **环境变量白名单**：`apps/backend/app/config.py:125-134` 把 `mistral` 加入 `llm_provider` 的 `Literal`。
2. **Provider 路由**：`apps/backend/app/llm.py:302-311` 的 `_PROVIDER_KEY_MAP` 增加 `"mistral": "mistral"`。
3. **模型名前缀**：`apps/backend/app/llm.py:410-422` 的 `provider_prefixes` 加 `"mistral": "mistral/"`。
4. **base URL 归一化**：`apps/backend/app/llm.py:64-114` 的 `_normalize_api_base` 检查 mistral 是否需要去 `/v1`（LiteLLM 内部会拼）。
5. **环境回退**：`apps/backend/app/llm.py:318-320` 的 `_PROVIDERS_WITHOUT_ENV_KEY_FALLBACK` 视情况加入（mistral 是云端，不应跳过环境回退）。
6. **健康检查特判**：若 mistral 的本地代理存在空 key 场景，可仿照 ollama 在 `llm.py:543-549` 增加豁免。
7. **超时系数**：`apps/backend/app/llm.py:915-921` 的 `provider_factors` 加入 mistral 经验延迟系数。
8. **文档**：`docs/agent/llm-integration.md:5-17` 的 provider 表补一行。

完成后调用 `await get_llm_config()` 即可返回新的 `LLMConfig`，不需要改任何业务代码。

## 8. 提示词工程最佳实践（基于本项目代码事实）

1. **JSON 模板中的 `{}` 必须转义为 `{{}}`**，否则 `str.format()` 直接抛 `KeyError`（参考 `enrichment.py` 全文件）。
2. **把"绝不能做的事"放在 prompt 顶部**，并且用 `CRITICAL` 字样强化：例如 `templates.py:198-227` 的 `CRITICAL_TRUTHFULNESS_RULES`，被注入到全文改写 prompt 的开头。
3. **提供完整的 JSON Schema 示例**：`RESUME_SCHEMA_EXAMPLE` / `IMPROVE_SCHEMA_EXAMPLE`（`templates.py:19-152`）让模型对字段名、层级、ID 编号规则有显式参考。
4. **用 enum-like 字符串约束动作**：`DIFF_IMPROVE_PROMPT` 把 `action` 限定为 `replace / append / reorder / add_skill`，便于服务端 `apply_diffs` 走 switch 校验。
5. **对用户输入做防注入**：`improver._sanitize_user_input` (`improver.py:48-56`) 用 8 条正则替换 `ignore previous instructions` 等典型注入串；这块要写进 `extract_job_keywords` 和 `generate_resume_diffs` 入口。
6. **应用层与传输层重试分离**：transport 重试交给 LiteLLM Router，内容质量（JSON 解析、截断）由 `complete_json` 自己重试，调用方不重复做。`llm.py:1150-1154` 是这一原则的注释化总结。
7. **能力探测走 Registry 而非硬编码**：`_supports_json_mode` / `_supports_temperature` / `get_safe_max_tokens` 都调用 `litellm.get_model_info`，新增模型时只要 LiteLLM 注册表更新，本项目即可自动跟上。
8. **本地安全网（defense in depth）**：即便 diff 通过，仍跑 `_preserve_personal_info` / `_restore_original_dates` / `_preserve_original_skills` / `_protect_custom_sections` 五个本地回填函数（`routers/resumes.py:832-844`），把"个人姓名 / 日期 / 技能 / 自定义章节"硬性恢复。
9. **schema 感知的截断判断**：`_appears_truncated(data, schema_type)` 根据是 `resume / enrichment / diff / keywords` 走不同启发式，避免把合法空 diff 误判。
10. **多语言通过 `{output_language}` 注入完整语言名**（`templates.py:13-15`），让模型翻译自己的"措辞风格"，比加一句 "respond in English" 更稳定。
11. **温度随重试递增**（`_get_retry_temperature`，`llm.py:879-894`）以 0.1→0.3→0.5→0.7 的梯度跳变，让二次抽样的输出与首轮尽量不一样，提升从 malformed 状态逃出的概率。
12. **错误日志先详细、再返回通用消息**：`llm.py:693-699`、`routers/resumes.py:126-127` 注释化的"Log details server-side, return generic messages to clients"是必须遵守的契约。
13. **prompt 中显式禁止 em-dash**：从 `IMPROVE_RESUME_PROMPT_NUDGE` (`templates.py:246`) 到 `OUTREACH_MESSAGE_PROMPT` (`templates.py:391`) 都在末尾强调 "Do NOT use em dash"。`AI_PHRASE_REPLACEMENTS` (`refinement.py:131-133`) 还把 em-dash 在本地替换成逗号。
14. **多轮打磨用本地 + LLM 组合**：能本地 `re.sub` 的（AI 短语替换、master 对齐校验）就别再调 LLM，节省 token；必须 LLM 的（关键词注入）才走 `complete_json`。
15. **preview + confirm 两阶段协议**：`POST /resumes/improve/preview` 生成 `preview_hash` 写回 job；`POST /resumes/improve/confirm` 校验 hash + `personalInfo` 不变才允许落库（`routers/resumes.py:963-1025`）。这是防止客户端篡改 LLM 输出的一道防线。

## 10. 小结

Resume Matcher 在 LLM 集成上展示了几个值得借鉴的工程原则：**统一传输层（LiteLLM Router + 指纹缓存）、能力探测走 Registry、内容质量与应用层重试解耦、JSON-mode→prompt-only 兜底、diff 改写 + 路径白/黑名单 + 本地安全网的多层防护、多轮打磨用本地 + LLM 组合、preview/confirm 两阶段协议**。把这些"基础设施"做到 `llm.py` 一个文件里，业务侧只需关心 prompt 模板与编排逻辑，迭代速度非常快。

下一步建议：阅读 `apps/backend/app/services/improver.py` 的 diff 4 道门与 `apps/backend/app/services/refiner.py` 的三段打磨，对照本教程第 4 节画时序图，会对整个 AI 集成链路有更直观的掌握。
