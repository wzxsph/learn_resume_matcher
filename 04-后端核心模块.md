## 一、数据库层 `apps/backend/app/database.py`

### 1.1 类结构

整个文件围绕一个 `Database` 类（`database.py:18-296`）展开，使用 TinyDB 4.8 + JSON 文件存储。类内部维护：

### 2.2 关键函数

#### `resolve_api_key`（`llm.py:323-349`）

```python
def resolve_api_key(stored: dict, provider: str) -> str:
    api_key = stored.get("api_key", "")
    if not api_key:
        api_keys = stored.get("api_keys", {})
        config_provider = _PROVIDER_KEY_MAP.get(provider, provider)
        env_default = (
            ""
            if provider in _PROVIDERS_WITHOUT_ENV_KEY_FALLBACK
            else settings.llm_api_key
        )
        api_key = api_keys.get(config_provider, env_default)
    return api_key
```

四个细节：
1. **顶层 `api_key` 优先**于 `api_keys[provider]`，保留向后兼容。
2. **本地 Provider 跳过 env 兜底**，避免付费 key 泄漏到本地 LLM。
3. `_PROVIDER_KEY_MAP`（`llm.py:302-311`）把内部 `gemini` 映射到配置文件中的 `google`。
4. 所有获取 key 的代码路径都必须走它；README/CLAUDE 反复强调这是"single source of truth"。

#### `get_llm_config`（`llm.py:352-400`）

读 `config.json` → 合并 Settings 默认值 → 处理 gpt-5 的"一次性迁移"（首次 `reasoning_effort` 缺失时自动写 `"minimal"`，但用户显式清空时不会再恢复）→ 返回 `LLMConfig`（Pydantic 模型）。

#### `get_model_name`（`llm.py:403-448`）

把"用户输入"转成 LiteLLM 期望的格式：

| Provider | 最终前缀示例 |
|----------|--------------|
| openai | `gpt-5-nano-2025-08-07`（无前缀） |
| openai_compatible | `openai/<model>`（使用 OpenAI 客户端） |
| ollama | `ollama_chat/<model>`（走 `/api/chat`） |
| openrouter | `openrouter/<model>` 或 `<model>`（已经带前缀时） |

如果模型名已经包含 `anthropic/`、`gemini/` 等已知前缀，则不再添加。

#### `get_router`（`llm.py:507-526`）

```python
def get_router(config: LLMConfig | None = None) -> tuple[Router, LLMConfig]:
    global _router, _router_config_key
    if config is None:
        config = get_llm_config()
    key = _config_fingerprint(config)
    with _router_lock:
        if _router is None or _router_config_key != key:
            _router = _build_router(config)
            _router_config_key = key
    return _router, config
```

线程锁 + 指纹缓存：只有当 `provider|model|hash(key)|api_base` 真正变化时才重建 Router。`_build_router`（`llm.py:472-504`）使用 `num_retries=3` 的 `RetryPolicy`，并显式 `disable_cooldowns=True`（单 deployment 时不需要冷却，否则一次失败会让整个后端停摆）。

#### `check_llm_health`（`llm.py:529-648`）

1. `ollama` 与 `openai_compatible` 允许 key 为空；其他 Provider 必须有 key，否则返回 `api_key_missing`。
2. 用一句话 `"Hi"` 调 `litellm.acompletion`，`max_tokens=64`、`timeout=30s`。
3. 失败时把异常信息按规则归类：`duplicate_v1_path`（`/v1/v1/`）、`not_found_404`、`html_response`、`health_check_failed`。
4. 调用 `_scrub_secrets`（`llm.py:286-299`）屏蔽可能出现在异常文本里的 `sk-...` / `AIza...` / `Bearer ...` 模式，避免在 Settings 页面被回读。
5. 探测时把 `test_prompt` 与 `model_output` 包装成 ` ```text ... ``` `，方便前端调试面板渲染。

#### `complete`（`llm.py:651-699`）

- 优先判断 `_supports_temperature`（`llm.py:827-876`）决定是否下发 `temperature`。
- 推理模型（deepseek-r1、qwq）的 `<think>...</think>` 由 `_strip_thinking_tags`（`llm.py:927-938`）剥离。
- 失败时记录详细日志，重新包装为通用 `ValueError` 返回给调用方。

#### `complete_json`（`llm.py:1026-1156`）

`complete_json` 是最复杂的对外函数。核心循环：

```python
for attempt in range(retries + 1):
    # 1) 调整 temperature（_get_retry_temperature 递增：0.1 → 0.3 → 0.5 → 0.7）
    # 2) 第一次失败后回退到 prompt-only JSON 模式（JSON-012）
    # 3) 调用 router.acompletion
    # 4) _extract_json 切出 JSON 字符串
    # 5) _appears_truncated(schema_type) 判断截断，命中则改写 prompt 重试
    # 6) 成功直接 return
```

错误分类：
- `json.JSONDecodeError` → 内容质量问题，重试 + 关闭 JSON 模式。
- `ValueError`（空响应 / 切不出 JSON） → 重试。
- 其他 `Exception` → **不重试**（Router 已经做过传输重试）。

| `get_safe_max_tokens` | `llm.py:736-778` | 把 `DEFAULT_JSON_MAX_TOKENS=8192` 截到模型上限 |

## 三、PDF 渲染 `apps/backend/app/pdf.py`

### 3.1 状态机

模块维护 4 个模块级变量（`pdf.py:26-30`）：

```python
_playwright = None
_browser: Optional[Browser] = None
_init_lock = asyncio.Lock()      # 防并发初始化
_subprocess_lock = asyncio.Lock()
_subprocess_supported = True      # 标记当前事件循环能否跑子进程
```

### 3.2 入口

```python
# apps/backend/app/pdf.py:251-310
async def render_resume_pdf(
    url: str,
    page_size: str = "A4",
    selector: str = ".resume-print",
    margins: Optional[dict] = None,
) -> bytes:
    ...
```

执行步骤：
1. `_resolve_pdf_format`（`pdf.py:54-59`）把 `"A4"` / `"LETTER"` 映射到 Playwright 接受的字符串。
2. `_resolve_pdf_margins`（`pdf.py:62-70`）统一为 `"{N}mm"`，避免 HTML padding 只影响首页。
3. 已有浏览器时直接渲染（`pdf.py:274-278`）。
4. 否则根据 `_subprocess_supported` 选择**主循环**（`async_playwright`）或**子线程**（`asyncio.to_thread` + 新事件循环）路径。
5. 最终调用 `_render_with_browser`（`pdf.py:146-157`），它会在新 page 上 `goto` → `wait_for_selector` → `document.fonts.ready` → `page.pdf`。

### 3.3 容错与错误分类

`_raise_playwright_error`（`pdf.py:209-227`）把 Playwright 异常映射成 `PDFRenderError`：

- `"Executable doesn't exist"` → 提示执行 `python -m playwright install chromium`。
- `"net::ERR_CONNECTION_REFUSED"` → 提示前端未启动 / `FRONTEND_BASE_URL` 不对。

`_find_chromium_executable`（`pdf.py:73-111`）依次扫描系统级 Chrome / Edge / Chromium / Snap / Flatpak，落到 Linux 桌面发行版的常见路径。

### 3.4 Windows 兼容

`main.py:11-12` 在 Windows 下设置 `WindowsProactorEventLoopPolicy`，让 Playwright 的子进程能在 asyncio 循环里运行。若循环仍不支持（`_loop_supports_subprocess` 返回 `False`），`render_resume_pdf` 会自动把渲染任务丢到 `asyncio.to_thread`（`pdf.py:198-206`）用 `ProactorEventLoop` 跑。

## 七、精修服务 `apps/backend/app/services/refiner.py`

`refiner.py`（702 行）实现"先关键词注入、再 AI 套话清理、最后 master 对齐"的多轮润色。核心入口是 `refine_resume`（`refiner.py:76-179`）：

1. 深度拷贝 `initial_tailored` 作为 `current`。
2. `analyze_keyword_gaps`（`refiner.py:181-231`）计算 `KeywordGapAnalysis`：JD 缺的关键词里只有"master 简历里也有"的才是 `injectable_keywords`，其余标记为 `non_injectable_keywords`（防止编造技能）。
3. `inject_keywords`（`refiner.py:452-...`）让 LLM 把可注入关键词自然嵌入当前简历。
4. `remove_ai_phrases`（`refiner.py:233-288`）纯本地操作：用 `AI_PHRASE_BLACKLIST` 替换 `AI_PHRASE_REPLACEMENTS`（来自 `prompts/refinement.py`）。
5. `validate_master_alignment`（`refiner.py:290-409`）跑 `AlignmentReport` 检查，必要时调 `fix_alignment_violations`（`refiner.py:515-563`）。
6. 循环直到 `passes_completed == max_refinement_passes` 或没有可改进之处。

`calculate_keyword_match`（`refiner.py:565-...`）计算首末两端与 JD 的匹配率，作为 API 响应里的 `RefinementStats`（`schemas/refinement.py:66-87`）。

### 9.3 改写预览 + 确认（curl 版）

```bash
# 1) 上传 JD
JOB_ID=$(curl -s -X POST http://localhost:8000/api/v1/jobs/upload \
  -H 'Content-Type: application/json' \
  -d '{"job_descriptions":["We are looking for a senior Python developer..."]}' \
  | jq -r '.job_id[0]')

# 2) 预览改写（不写入）
PREVIEW=$(curl -s -X POST http://localhost:8000/api/v1/resumes/improve/preview \
  -H 'Content-Type: application/json' \
  -d "{\"resume_id\":\"$RESUME_ID\",\"job_id\":\"$JOB_ID\"}")
echo "$PREVIEW" | jq '.data.preview_hash, .data.resume_preview.personalInfo.name'

# 3) 确认写入
curl -s -X POST http://localhost:8000/api/v1/resumes/improve/confirm \
  -H 'Content-Type: application/json' \
  -d "$(jq -n --arg rid $RESUME_ID --arg jid $JOB_ID --arg ph "$PREVIEW_HASH" \
        '{resume_id:$rid, job_id:$jid, improved_data:<从 preview 复制>, preview_hash:$ph}')"
```

> `preview_hash` 是 `_hash_improved_data`（`resumes.py:94-104`）算出的 SHA-256，后端在 confirm 时会重新计算并比对，校验失败则返回 400。

## 十、调试小贴士

- **LiteLLM 静默**：默认 `log_llm=WARNING`（`config.py:138`）。改成 `DEBUG` 并重启，可以看到每条请求的 provider/model/usage。
- **Router 重建日志**：切换 Provider/模型/Key 时会打印 `"LiteLLM Router rebuilt for ..."`（`llm.py:523`），方便确认缓存确实失效。
- **PDF 渲染失败**：`PDFRenderError` 异常信息里会直接给出 `python -m playwright install chromium` 命令，照抄即可。
- **Master 简历异常**：当 master 状态卡在 `failed` 或 `processing`，下次上传会自动晋升新 master（`database.py:119-127`）。
- **Schema 懒迁移**：旧数据若缺 `sectionMeta`，`GET /resumes?resume_id=...` 会自动用 `DEFAULT_SECTION_META`（`models.py:268-323`）填充，无需手动迁移。

## 十一、小结

- `database.py` 用类级 `asyncio.Lock` 守住"恰好一个 master 简历"的不变量。
- `llm.py` 是整个后端的"复杂度中心"：Router 缓存、应用层重试、schema 感知截断、provider 能力探测都在它里面。
- `pdf.py` 通过懒加载 + 平台分支处理让 Chromium 在所有桌面平台都能跑。
- `config.py` 的 16 个端点可以"在线换 Provider / 改 Key / 自定义 Feature Prompt"，并且所有写操作都通过 `invalidate_config_cache()` 让其他模块感知。
- `improver.py` + `refiner.py` 共同构成"diff 改写 + 多轮润色"双层防护；`resumes.py` 的 preview/confirm 流程则把 LLM 不可控因素挡在 master 之外。

到此后端核心模块的逐函数拆解就完成了。下一步可结合 [docs/agent/architecture/backend-guide.md](https://github.com/srbhr/Resume-Matcher/blob/main/docs/agent/architecture/backend-guide.md) 与 `apps/backend/tests/` 跑通端到端测试，进一步体会各接口的边界条件。
