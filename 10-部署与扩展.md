# 10 - 部署与扩展

> 本章面向需要把 Resume Matcher 部署上线、调整环境配置、添加新 provider / 模板 / 语言的开发者。我们将逐项拆解本地开发、Docker 镜像、PDF 渲染流水线，并给出扩展点与性能调优建议。

---

## 1. 本地开发部署

### 1.1 后端

后端使用 `uv` 管理依赖（`pyproject.toml` 与 `requirements.txt`），Python 3.13+，核心栈：FastAPI、Pydantic v2、TinyDB、LiteLLM、markitdown、Playwright。

```bash
cd apps/backend
uv sync                                              # 创建 .venv 并安装依赖
uv run uvicorn app.main:app --reload --port 8000     # 开发服务器
uv run app                                           # 等价于 app.main:main，支持 HOST/PORT/RELOAD
uv run playwright install chromium                   # 一次性：下载 Chromium
```

API 文档自动暴露在 `http://localhost:8000/docs`（Swagger UI）和 `/redoc`。

### 1.2 前端

```bash
cd apps/frontend
npm install
npm run dev      # next dev --turbopack，:3000
npm run build    # next build（运行 tsc，i18n 结构错位时构建失败）
npm run start
npm run lint     # eslint .
npm run format   # prettier --write .
npm run test     # vitest run
```

> 前端不能直接 `fetch('http://localhost:8000')`。所有后端请求都走 `lib/api/`（`apps/frontend/CLAUDE.md:67-86`），通过 `next.config.ts` 的 `rewrites` 把 `/api/*` 代理到后端的 `BACKEND_ORIGIN`。不要创建 `app/api/` 路由，会覆盖代理。

### 1.3 开发期间的依赖关系

- 前端在 :3000 监听浏览器访问，把 `/api/*` 代理到 :8000。
- 后端在生成 PDF 时再把请求转回 `{FRONTEND_BASE_URL}/print/*`，由 Playwright 渲染（见 §4）。
- 因此必须**同时启动前后端**，否则 PDF 会拿到 503 错误（`apps/backend/app/pdf.py:218-226`）。

## 2. Docker 部署

### 2.1 镜像结构（`Dockerfile`）

Dockerfile 是多阶段构建：

#### Stage 1：frontend-builder（`Dockerfile:7-27`）

```dockerfile
FROM node:22-bookworm AS frontend-builder
ARG NEXT_PUBLIC_API_URL=/
COPY apps/frontend/package*.json ./
RUN npm ci
COPY apps/frontend/ ./
RUN npm run build
```

`NEXT_PUBLIC_API_URL` 允许构建时自定义，默认 `/` 表示请求经由 Next.js rewrite 同源转发。`NEXT_TELEMETRY_DISABLED=1` 关闭遥测。

#### Stage 2：最终镜像（`Dockerfile:32-127`）

```dockerfile
FROM python:3.13-slim-bookworm
```

- 安装系统级 Playwright 依赖：`libnss3 / libnspr4 / libatk1.0-0 / libatk-bridge2.0-0 / libcups2 / libdrm2 / libxkbcommon0 / libxcomposite1 / libxdamage1 / libxfixes3 / libxrandr2 / libgbm1 / libasound2 / libpango-1.0-0 / libcairo2 / libatspi2.0-0 / libgtk-3-0`（`Dockerfile:43-64`）。
- 从 `frontend-builder` 复制 Node.js 运行库（`/usr/local/bin/node`），让 `server.js` 可以启动。
- 复制 `apps/backend` 并 `pip install .`。
- 复制前端 standalone 输出：`.next/standalone`、`/.next/static`、`/public`（`Dockerfile:87-90`）。
- 创建非 root 用户 `appuser:1000` 并 `chown` 整个 `/app`（`Dockerfile:104-108`），提升安全基线。
- 以 `appuser` 身份执行 `python -m playwright install chromium`（`Dockerfile:111`），保证浏览器落在正确路径。
- 暴露 3000、声明 `/app/backend/data` 为 volume、健康检查 `127.0.0.1:8000/api/v1/health`（`Dockerfile:114-127`）。

> 健康检查走的是**容器内**的 :8000，**不**依赖宿主机端口映射。

### 2.2 docker-compose.yml 解析

```yaml
services:
  resume-matcher:
    image: ghcr.io/srbhr/resume-matcher
    build: .
    container_name: resume-matcher
    ports:
      - "${PORT:-3000}:3000"
    volumes:
      - resume-data:/app/backend/data
    environment:
      - FRONTEND_BASE_URL=${FRONTEND_BASE_URL:-http://localhost:3000}
      - LOG_LEVEL=${LOG_LEVEL:-INFO}
      - LOG_LLM=${LOG_LLM:-WARNING}
      - LLM_PROVIDER=${LLM_PROVIDER:-openai}
      - LLM_MODEL=${LLM_MODEL:-}
      - LLM_API_KEY=${LLM_API_KEY:-}
      - LLM_API_BASE=${LLM_API_BASE:-}
    restart: unless-stopped

volumes:
  resume-data:
    driver: local
```

要点（`docker-compose.yml:1-42`）：

- **单端口暴露**：宿主机 `${PORT:-3000}` → 容器 :3000；后端 :8000 仅在容器内部可访问，由前端的 `rewrites` 转发。
- **数据持久化**：`resume-data` 命名卷挂到 `/app/backend/data`，数据库与上传文件不会因容器重建丢失。
- **环境变量**：所有 LLM 与日志相关变量都从宿主机 `.env` 注入，未提供时取默认值。
- **Secret File 模式**：可使用 `LLM_API_KEY_FILE` 指向挂载的 secret，行为类似 Postgres 的 `*_FILE` 约定（`docker-compose.yml:25-27`，由 `docker/start.sh:65-88` 解析）。
- **注释掉的 CORS**：默认会自动把 `FRONTEND_BASE_URL` 加入白名单。

### 2.3 启动脚本（`docker/start.sh`）

启动脚本完成三件事：

1. **环境解析**：把 `LOG_LEVEL` / `LLM_PROVIDER` / `LLM_API_KEY[_FILE]` 等转换为真正的环境变量并校验（`docker/start.sh:65-88`）。
2. **日志级别归一化**：`normalize_log_level` 拒绝非法值并打印警告（`docker/start.sh:90-104`）。
3. **进程编排**：
   - 后端：uvicorn 监听 :8000，30 秒内未就绪则报错退出。
   - 前端：`node server.js "$@"` 监听 :3000。
   - 任一进程退出即触发 `cleanup` 杀掉另一个（`docker/start.sh:117-131, 203-256`）。

启动时还会按需安装 Chromium（`docker/start.sh:191-201`），但镜像里已经预装，正常情况只走快速路径。

### 2.4 启动方式

```bash
# 1) 创建 .env
cat > .env <<'EOF'
LLM_PROVIDER=openai
LLM_MODEL=gpt-5-nano-2025-08-07
LLM_API_KEY=sk-your-key
FRONTEND_BASE_URL=http://localhost:3000
LOG_LEVEL=INFO
EOF

# 2) 启动
docker compose up -d
docker compose logs -f

# 3) 升级 / 重建
docker compose build --pull
docker compose up -d
```

## 3. 生产环境部署注意事项

### 3.1 反向代理与域名

`FRONTEND_BASE_URL` 必须写成**浏览器可访问的公网/内网 URL**，因为 Playwright 通过这个 URL 抓取 `/print/*`。如果前端挂在 `https://resume.example.com`，设置：

```env
FRONTEND_BASE_URL=https://resume.example.com
```

同时确认后端的 `CORS_ORIGINS` 包含该域名（默认值在 `apps/backend/.env.example` 中）。

### 3.2 HTTPS 与 Cookie

- 反向代理（Nginx / Caddy / Traefik）终止 TLS，前端始终走 443。
- 如部署在内网并启用 `secure` cookie，确保代理转发 `X-Forwarded-Proto: https`。

### 3.3 数据与备份

- **数据库与上传**都落在 `/app/backend/data`，对应 `resume-data` 卷。定期 `docker run --rm -v resume-data:/data -v $(pwd):/backup alpine tar czf /backup/resume-data.tgz /data`。
- `.env` **不要**提交到仓库，建议用 secret manager。

### 3.4 单 worker 假设

后端的 `asyncio.Lock` 与 `config_cache` 都假设**单 uvicorn worker**。生产部署请使用单进程（`--workers 1`），如需横向扩展，需要先改造 `config_cache` 与 master lock（见 `apps/backend/CLAUDE.md:148`）。

### 3.5 健康检查

`Dockerfile:123-124` 默认指向 `/api/v1/health`（不调用 LLM，仅 liveness）。如需更全面的状态，请额外探活 `/api/v1/status`（包含 LLM health 与 DB 统计）。

### 3.6 日志与告警

- `LOG_LEVEL` 控制应用日志；`LOG_LLM` 控制 LiteLLM 详细输出。
- 生产建议 `LOG_LEVEL=INFO`、`LOG_LLM=WARNING`。
- **切勿**把 `LOG_LLM=DEBUG` 长期打开——会打印明文 API key（`docker/start.sh:176-178` 提示）。

## 5. PDF 渲染流水线

PDF 由后端驱动，**前端是模板源，后端是渲染器**。流水线如 `docs/agent/design/pdf-template-guide.md:6-15` 所述。

### 5.1 入口

`apps/backend/app/pdf.py:251-310` 的 `render_resume_pdf()`：

```python
await page.goto(url, wait_until="networkidle")
await page.wait_for_selector(selector)            # 默认 ".resume-print"
await page.evaluate("document.fonts.ready")
return await page.pdf(
    format=pdf_format,                           # A4 / Letter
    print_background=True,
    margin=pdf_margins,                          # 来自 *_BASE_URL 的 host 决定
)
```

- `url` 由后端构造：`{FRONTEND_BASE_URL}/print/resumes/{id}?template=...&pageSize=...&lang=...&...`。
- 等待选择器确保 React 树已挂载；`document.fonts.ready` 等待 web font 加载。
- margins 来自 `TemplateSettings.margins`（5-25mm），通过 Playwright 注入 PDF 而不是 HTML 内边距，**保证多页一致**（`docs/agent/design/print-pdf-design-spec.md:35-39`）。

### 5.2 启动路径与回退

`apps/backend/app/pdf.py:33-51` 懒加载浏览器，`init_pdf_renderer` 使用 `asyncio.Lock` 防并发初始化。如果主事件循环不支持子进程（Windows 旧版 asyncio），会退化到 `asyncio.to_thread` 在新 loop 中跑同步实现（`apps/backend/app/pdf.py:160-206, 280-302`）。

### 5.3 浏览器定位策略

`apps/backend/app/pdf.py:73-111` 的 `_find_chromium_executable` 在以下位置查找系统 Chrome/Chromium/Edge（按平台分支）：

- Windows：`Program Files\Google\Chrome\Application\chrome.exe`、`msedge.exe`。
- macOS：`/Applications/Google Chrome.app/...`、`/Applications/Chromium.app/...`。
- Linux：`/usr/bin/google-chrome`、`chromium`、`microsoft-edge`、Snap、Flatpak 等。

主路径失败时 fallback 到系统安装的浏览器（`apps/backend/app/pdf.py:114-126`）。

### 5.4 错误分类

`apps/backend/app/pdf.py:209-227` 对 Playwright 抛错做友好分类：

- `Executable doesn't exist` → 提示运行 `python -m playwright install chromium`。
- `net::ERR_CONNECTION_REFUSED` → 提示检查前端是否启动、`FRONTEND_BASE_URL` 是否正确。
- 其它错误透传为 `PDFRenderError("PDF rendering failed: ...")`。

### 5.5 打印路由

| 路由 | 入口文件 | 关键选择器 |
|------|----------|------------|
| `/print/resumes/[id]` | `apps/frontend/app/print/resumes/[id]/page.tsx:1` | `.resume-print` |
| `/print/cover-letter/[id]` | `apps/frontend/app/print/cover-letter/[id]/page.tsx:69` | `.cover-letter-print` |

二者均为 Server Component，从 `searchParams` 读取 `template` / `pageSize` / `lang` 等参数，再通过 `lib/i18n/server.ts:translate()` 本地化标题。

> **关键 CSS 规则**（`docs/agent/design/pdf-template-guide.md:39-50`）：`@media print` 必须把 `body *` 置为 `visibility: hidden`，再把 `.resume-print *` / `.cover-letter-print *` 置为 `visible`，否则 PDF 会全白。

### 5.6 分页逻辑

`apps/frontend/components/preview/use-pagination.ts:43-175` 计算分页：

- 用 `ResizeObserver` + `MutationObserver` 监听内容变化，150 ms debounce 重算。
- 找出所有 `.resume-item` 与 `[data-no-break]`，构建不可分页集合。
- 章节标题自动扩展到第一个内容元素，避免孤立。
- 算法约束：分页点不能把一个 item 拆开，且**至少占满 50% 页面**（`use-pagination.ts:130-144`）。

### 5.7 排版规则

来自 `docs/agent/design/print-pdf-design-spec.md:41-46`：

- `.resume-item` 永不跨页。
- 章节标题不孤立。
- 单页至少 50% 填充后再换页。

## 6. 添加新 LLM Provider

`apps/backend/app/llm.py` 通过 LiteLLM 抽象多 provider。新增 provider 一般无需改 LLM 层，只需要在 Settings 页面 / 环境变量中配置即可。但若需要把 provider 注册为 **官方可选项**，可按以下步骤：

1. **确认 LiteLLM 支持**：在 LiteLLM 文档中查找 provider 标识（如 `bedrock`、`cohere`）。
2. **在 `apps/backend/app/config.py`** 中将 provider 名字加入 `LLMProvider` 字面量联合（如有显式枚举）。
3. **在 `apps/backend/app/llm.py:get_model_name()`** 中添加 provider → LiteLLM model prefix 映射（如 `bedrock` → `bedrock/...`）。
4. **如果走非标准 base URL**：在 `_normalize_api_base()` 中处理尾随 `/v1` 重复问题。
5. **添加默认模型**（可选）：在 `apps/frontend/lib/api/config.ts:PROVIDER_INFO` 中追加。
6. **更新 `docker-compose.yml`** 注释（`docker-compose.yml:22-23`）与 `apps/backend/.env.example`，方便用户发现。
7. **测试**：使用 `POST /api/v1/config/llm-test`（`apps/backend/app/routers/config.py`）做活体检。

特殊情形：

- **`openai_compatible`**（llama.cpp / vLLM / LM Studio）：在 `apps/backend/app/llm.py` 故意**忽略**环境变量 `LLM_API_KEY` fallback，避免付费 key 误传到本地服务（`apps/backend/CLAUDE.md:103`）。
- **Ollama**：在 Docker 部署中使用 `host.docker.internal:11434` 访问宿主机 Ollama（`docker-compose.yml:29-30`）。
- **支持 `reasoning_effort`** 的模型：LiteLLM 会在不支持的 provider 上自动 drop（`litellm.drop_params = True`，`apps/backend/CLAUDE.md:96`）。

## 10. 常见问题排查

| 症状 | 排查要点 |
|------|---------|
| PDF 导出空白 | 检查 `globals.css` 是否包含 `@media print` 的 `.resume-print` 可见性规则（`docs/agent/design/pdf-template-guide.md:39-50`） |
| PDF 报 `ERR_CONNECTION_REFUSED` | 确认前端在 `${FRONTEND_BASE_URL}` 上启动；Docker 内确认 `FRONTEND_BASE_URL` 与暴露端口一致（`apps/backend/app/pdf.py:218-226`） |
| `next build` 失败但 `next dev` 正常 | 必有 locale JSON 缺失 key，类型推导报错（`apps/frontend/CLAUDE.md:115-122`） |
| `playwright install` 失败 | 镜像内已预装，本地若失败可手动 `uv run playwright install chromium`（`apps/backend/CLAUDE.md:30`） |
| Docker 内 LLM 调不通 | Ollama 用 `host.docker.internal`（`docker-compose.yml:29-30`）；检查 `LLM_API_KEY` 是否被 `LLM_API_KEY_FILE` 覆盖（`docker/start.sh:69-72`） |
| 日志出现明文 key | `LOG_LLM=DEBUG` 会打印。生产请保持 `WARNING`（`docker/start.sh:176-178`） |
| API 返回 401/403 | 确认 `LLM_API_KEY` 是否被后端从 `config.json` 覆盖；写 config 后需 `invalidate_config_cache()`（`apps/backend/CLAUDE.md:144`） |
| 改善后日期缺月 | LLM 丢精度是已知问题，后端用 `restore_dates_from_markdown` + `_restore_original_dates` 修复（`apps/backend/CLAUDE.md:147`） |
| Improve 返回 400 | 必须先调用 `/improve/preview` 获取 `preview_hash`，`/improve/confirm` 才会接受（`apps/backend/CLAUDE.md:151`） |
| 容器权限报错 | 镜像以 `appuser:1000` 运行，宿主挂载卷需 `chown -R 1000:1000` 或使用命名卷（`docker-compose.yml:7-9`） |
| 主语种选择器无新语言 | 检查 `i18n/config.ts` 的 `locales` 数组（`apps/frontend/i18n/config.ts:5`） |

## 11. 性能调优建议

### 11.1 后端

- **超时**：`asyncio.wait_for` 默认 240s（`apps/backend/CLAUDE.md:95`）。如使用更慢的本地模型，可提高 `LLM_API_BASE` 调优 `get_safe_max_tokens`。
- **JSON 模式**：`complete_json()` 自动 JSON-mode→prompt fallback。给 provider 加上 `num_retries=3`（已在 `get_router` 中默认配置，`apps/backend/CLAUDE.md:91`）。
- **缓存**：`config_cache` TTL 5 分钟；如需立刻生效调用 `invalidate_config_cache()`（写 `config.json` 路由自动做）。
- **PDF 并发**：`_init_lock` 保证单浏览器实例多请求复用。如需更高并发可考虑多 worker + 多浏览器，但要注意 `/app/backend/data` 是文件存储，并发写需自行加锁（推荐外置 Postgres）。
- **Prompt 长度**：`PARSE_RESUME_PROMPT` 嵌入整个 schema 体积较大（`apps/backend/CLAUDE.md:79-80`）。如不需特定字段，可在 `RESUME_SCHEMA_EXAMPLE` 中裁剪。

### 11.2 前端

- **包体优化**：`lucide-react` 用深路径 import（如 `lucide-react/dist/esm/icons/x`），并由 `next.config.ts` 的 `optimizePackageImports` 进一步摇树（`apps/frontend/CLAUDE.md:178-180`）。
- **超时**：`apiFetch` 默认 240 s（`apps/frontend/CLAUDE.md:84-85`），与后端匹配。
- **打印路由**：`print/*` 保持 Server Component 避免 hydration 成本（`apps/frontend/CLAUDE.md:182-183`）。
- **i18n 大小**：5 个语言 × ~35 KB JSON 全部静态 import，初始 bundle 不可避免。若需按需加载，改成 `next/dynamic({ ssr: false })` 加载特定语言，但会牺牲 SSR。
- **预渲染**：`PaginatedPreview` 用隐藏的 `measurementRef` + `ResizeObserver` + `MutationObserver`（`use-pagination.ts:188-218`）做客户端分页，权衡：避免 SSR 时算不准，但内容频繁变更会触发重算（150 ms debounce 已缓解）。

### 11.3 Docker

- **多阶段构建** 显著减小最终镜像体积。
- **Playwright 系统依赖** 已全部内置，无需 `--with-deps`。
- **Chromium 冷启动**：容器内预装到 `/home/appuser/.cache/ms-playwright`（`docker/start.sh:193`），首次启动即可用。

### 11.4 观测

- `/api/v1/status` 提供 LLM health（30 min 缓存）+ DB 统计（5 min 缓存），由 `StatusCacheProvider` 同步到前端（`apps/frontend/CLAUDE.md:96-98`）。
- 应用日志通过 `LOG_LEVEL` 控制；LiteLLM 通过 `LOG_LLM`。
- 建议把容器 stdout 接到 Loki/CloudWatch/GCP Logging，匹配 alert 规则（如 PDF 错误率 > 1%、健康检查失败）。
